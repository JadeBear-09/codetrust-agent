# Submission checklist

## Eligibility and access

- [ ] Confirm invite-only Round 1 email from `support@hackerearth.com`.
- [ ] Confirm HackerEarth profile, resume, experience, and contact details are current.
- [ ] Save challenge timing in Asia/Kolkata.
- [ ] Read assessment email rules before starting; timed challenge cannot be paused.

## Repository

- [ ] `main` is clean and CI green.
- [ ] No API key, token, `.env`, customer data, or private prompt in git history.
- [ ] README setup works from fresh clone.
- [ ] `make test`, `make lint`, `make demo-offline`, and `make proof` pass as intended.
- [ ] Docker image builds and health endpoint responds.
- [ ] Final release tag created after event problem is solved.

## Demo

- [x] Analyze public GnuCash PR and confirm `BLOCK 100/100`.
- [x] Confirm recorded maintainer outcome is closed unmerged as out of scope.
- [x] Analyze private CodeTrust PR and confirm `BLOCK 94/100`.
- [ ] Record public policy-match flow without claiming blind prediction.
- [ ] Show exact duplicate-payment and async-blocking evidence.
- [ ] Finish spoken flow under five minutes.
- [ ] Record 90-second backup video.
- [ ] Keep offline report and screenshots ready.

## Submission form

- [ ] Project name and one-line pitch.
- [ ] Problem, affected user, and measurable cost.
- [ ] Solution workflow and differentiation.
- [ ] Architecture diagram.
- [ ] Repository link and setup instructions.
- [ ] Demo video link.
- [ ] Measured results, not placeholder claims.
- [ ] Limitations and responsible-AI boundary.
- [ ] Team size shown as one.

## Judge defense

- [ ] Explain every source file.
- [ ] Explain why model cannot approve change.
- [ ] Explain false-positive and false-negative handling.
- [ ] Explain production isolation and secret controls.
- [ ] Explain why existing linters are insufficient.
- [ ] Explain how solution changes for actual DTDL problem statement.
